## Subscription Flow

```
User downloads app
       ↓
Creates account (email + password)
       ↓
Chooses plan: Monthly (£9.99) or Yearly (£95.88)
       ↓
Enters payment details (card required)
       ↓
Subscription active - 14 days FREE
       ↓
Day 15: Card charged automatically
       ↓
Recurring billing (monthly or annually)
```

### Cancellation

| Timing | Outcome |
|--------|---------|
| Cancel before Day 14 | No charge, data read-only for 30 days |
| Cancel after billing starts | Access until end of paid period |
| After access ends | Data read-only for 30 days, then deleted |

---

## Why No Free Tier

| Concern | How Subscription-First Solves It |
|---------|----------------------------------|
| Competitor abuse | Card required blocks casual inspection |
| Resource abuse | No forever-free users draining storage |
| Multi-account abuse | Payment friction prevents spam accounts |
| Cost protection | Every user pays or leaves |
| Quality users | Only serious professionals commit |

## Storage Management

### Industry Approach

| Method | Description |
|--------|-------------|
| **Soft limits** | No hard limit, contact if excessive |
| **Per-user allocation** | e.g., 5GB per user |
| **Unlimited (fair use)** | Terms say "reasonable use" |
| **Fixed quota** | e.g., 25GB total (SnagSnapper) |

**Note:** Bandwidth is NOT measured in the industry - only storage.

### Why Not Measure Bandwidth

| Reason | Explanation |
|--------|-------------|
| Confusing to users | "What's a GB of bandwidth?" |
| Unpredictable | User can't control sync frequency |
| Sync is incremental | Only changed data transfers |
| Photos upload once | Not re-downloaded constantly |

### SnagSnapper Storage Tracking

**Technical Implementation:**

```
On Image Upload:
1. Compress image (target ~300KB)
2. Upload to Firebase Storage
3. Add file size to user's storageUsed field
4. Check: if storageUsed > limit → block upload

On Image Delete:
1. Delete from Firebase Storage
2. Subtract file size from storageUsed

Weekly Reconciliation (Cloud Function):
1. List all files in user's storage path
2. Sum actual sizes
3. Update storageUsed (corrects any drift)
```

**Data Model:**

```dart
// In AppUser/Profile
int storageUsedBytes;       // Current usage
int storageLimitBytes;      // 25GB = 26,843,545,600 bytes
DateTime? storageCheckedAt; // Last reconciliation
```

### Realistic Storage Usage

| User Type | Calculation | Total Storage |
|-----------|-------------|---------------|
| Light | 2 sites × 50 snags × 2 photos × 300KB | ~60MB |
| Typical | 5 sites × 100 snags × 3 photos × 300KB | ~450MB |
| Heavy | 10 sites × 300 snags × 4 photos × 300KB | ~3.6GB |
| Maximum | 10 sites × 500 snags × 4 photos × 300KB | ~6GB |

**25GB limit covers even the heaviest professional use.**

## Permission System

### Access Levels

| Level | Description |
|-------|-------------|
| **VIEW** | Read-only access to site and all snags |
| **WORKING** | Work on assigned snags only |
| **CONTRIBUTOR** | Create snags, work on assigned |
| **OWNER** | Full control (not a permission, identified by email) |

### Configurable Settings (Per-Site)

| Setting | Default | Description |
|---------|---------|-------------|
| `workingCanSeeAllSnags` | `true` | WORKING members see all snags (but only edit assigned) |
| `contributorCanEditOthers` | `false` | CONTRIBUTOR can edit snags created by others |
