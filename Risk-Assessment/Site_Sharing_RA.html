2.5 Data Integrity (Version Sync) - Detailed Explanation

The Problem:

When a site is shared with collaborators, they need to know when the site has been updated. Without version tracking:

  • Collaborator's app would need to fetch the full site document from Firebase on every sync
  • This causes unnecessary Firestore reads and costs
  • No efficient way to detect "has anything changed?"

The Solution:

CF stores { ownerUID, version } in each collaborator's shared_access entry.

Code Implementation:

1. Version detection (lines 266-268):

const siteVersion = afterData.firebaseVersion || 0;
const beforeVersion = beforeData.firebaseVersion || 0;
const versionChanged = siteVersion !== beforeVersion;

2. New collaborator - gets both ownerUID and version (lines 307-312):

[`sites.${siteId}`]: { ownerUID, version: siteVersion }

3. Existing collaborator - version updated when site changes (lines 320-332):

} else if (versionChanged) {
  // Existing user - update version only
  updatePromises.push(
    accessRef.update({
      [`sites.${siteId}`]: { ownerUID, version: siteVersion }
    })
  );
}

Client-Side Sync Flow:

  1. Collaborator's app reads their shared_access document
  2. Compares local version with remote version using < (less than)
  3. If localVersion < remoteVersion → client is behind, redownload site
  4. If localVersion >= remoteVersion → already up to date, skip fetch
// Client-side comparison logic
if (localVersion < shared_access.sites[siteId].version) {
  // Local is behind - redownload site from Firebase
  fetchSiteDocument(siteId, ownerUID);
}

Why This Matters:

  • Collaborator only reads 1 small document (shared_access) to check ALL shared sites
  • Only fetches full site document when version indicates changes
  • Saves Firestore read costs and bandwidth
